# 博主追踪模块性能优化方案

## 一、当前性能瓶颈定位

### 1.1 现有实现的请求链路

追踪一个 B 站博主的完整请求流程如下：

| 步骤 | 请求内容 | 耗时估算 | 风险点 |
|------|----------|----------|--------|
| ① | 初始化 Session，访问 bilibili.com 获取 Cookie | 5-15 秒 | 可能失败，需重试 |
| ② | 调用 space/acc/info API 获取博主信息 | 3-10 秒 | 可能返回 -799 限流 |
| ③ | 调用 space/arc/search API 获取视频列表 | 5-15 秒 | 高概率返回 -799/-412/-352 |
| ④ | 降级到搜索 API（用 UP 主名搜索） | 5-15 秒 | 搜索 API 同样可能限流 |
| ⑤ | AI 批量生成视频概要 | 5-15 秒 | 网络延迟+AI 处理时间 |

**核心问题**：每步之间有强制等待机制，代码中设置了：
- `_MIN_REQUEST_INTERVAL = 2.5 秒`（基础间隔）
- 连续失败后自适应加长（最高 10 秒）
- 博主间随机等待 2-4 秒
- 批次间随机等待 5-10 秒

### 1.2 实际耗时测算

假设追踪 10 个 B 站博主（按当前配置）：
- 每个博主平均需要 3-4 次 API 请求
- 每次请求平均等待 2.5 秒（基础）+ 网络延迟 2-5 秒
- 博主间等待平均 3 秒
- **估算总耗时：10 × (3 × 7.5 + 3) = 约 255 秒 ≈ 4.25 分钟**

如果遇到限流重试，可能超过 8-10 分钟。

## 二、yupi-hot-monitor 的 B 站 API 方案分析

### 2.1 核心 API 调用方式

yupi-hot-monitor 使用了 **2 个 B 站公开 API**：

#### API 1: 搜索视频
```
GET https://api.bilibili.com/x/web-interface/search/type
参数：keyword=关键词，search_type=video, order=pubdate, pagesize=20
```

#### API 2: 获取用户视频列表
```
GET https://api.bilibili.com/x/space/arc/search
参数：mid=用户 ID, pn=页码，ps=10, order=pubdate
```

### 2.2 关键优化策略

| 策略 | 实现方式 | 效果 |
|------|----------|------|
| **buvid3 Cookie 自动生成** | 使用 `crypto.randomUUID()` 生成 | 无需访问首页获取 Cookie，减少 1 次请求 |
| **宽松限流** | 全局 2 秒 间隔 | 比当前的 2.5 秒 + 自适应机制更宽松 |
| **单 API 直调** | 直接调用目标 API | 无多级降级逻辑，路径最短 |
| **错误快速失败** | 返回空数组而非重试 | 避免阻塞流程 |

| 维度 | free-video-downloader (现有) | yupi-hot-monitor |
|------|------------------------------|------------------|
| **Cookie 获取** | 访问首页获取真实 Cookie（5-15 秒） | 随机 UUID 生成（瞬间） |

| **限流策略** | 2.5 秒基础 + 自适应惩罚（最高 10 秒） | 固定 2 秒 |
| **错误处理** | 多次重试 | 快速失败 |

| **单博主耗时** | 约 20-40 秒 | 约 3-6 秒 |
| **10 个博主总耗时** | 约 4-10 分钟 | 约 30-60 秒 |
| **稳定性** | 高（多级降级） | 中（依赖 buvid3 有效） |
| **反爬风险** | 中（访问模式较自然） | 低（buvid3 生成方式被广泛使用） |

## 四、优化方案可行性评估

### 4.1 ✅ 可采纳的优化策略

| # | 优化项 | 可行性 | 预期效果 | 实施难度 |
|---|--------|--------|----------|----------|
| 1 | **Cookie 自动生成**：用随机 UUID 替代首页访问 | **高** | 每博主节省 5-15 秒 | 低 |
| 2 | **减少 API 降级**：保留 2 级降级即可 | **高** | 每博主节省 5-10 秒 | 中 |
| 3 | **缩短基础限流**：从 2.5 秒降至 1.5-2 秒 | **高** | 总体提速 20-30% | 低 |
| 4 | **减少重试次数**：从 3 次降至 2 次 | **高** | 失败场景提速 30% | 低 |

### 4.2 ⚠️ 需要谨慎评估的策略

| # | 优化项 | 风险 | 建议 |
|---|--------|------|------|
| 5 | **完全移除自适应限流** | 可能触发 B 站封禁 | ❌ 不建议移除，但可以缩短惩罚间隔 |
| 6 | **移除 AI 概要生成** | 会影响用户体验 | ❌ 这是核心功能，不能移除。但可以优化为并行调用 |

### 4.3 ❌ 不建议的优化

| # | 优化项 | 原因 |
|---|--------|------|
| 7 | 完全移除错误重试 | B 站 API 不稳定，需要容错 |
| 8 | 并发请求多个博主 | 极易触发封禁 |

## 五、推荐的优化方案（分阶段实施）

### 🟢 第一阶段：低风险优化（推荐立即实施）

**预期效果**：耗时从 4-10 分钟 降至 **1.5-3 分钟**（提速约 60-70%）

1. **Cookie 自动生成**：参考 yupi-hot-monitor，使用随机 UUID 生成 buvid3
2. **缩短基础限流**：从 2.5 秒 降至 1.5 秒
3. **减少降级层级**：保留 space/acc/info → 搜索 API 两级降级
4. **减少重试次数**：space API 从 3 次重试降至 2 次

### 🟡 第二阶段：中风险优化（第一阶段验证后实施）

**预期效果**：耗时从 1.5-3 分钟 降至 **40-90 秒**（再提速约 50%）

1. **AI 概要生成优化**：改为异步并行调用，不阻塞主流程
2. **自适应限流优化**：缩短惩罚间隔（从 +1 秒 改为 +0.5 秒）
3. **博主间等待缩短**：从 2-4 秒 降至 1-2 秒

### 🔴 第三阶段：架构优化（长期规划）

**预期效果**：进一步提升稳定性和用户体验

1. **引入缓存机制**：博主信息缓存 10 分钟，避免重复请求
2. **增量更新模式**：只获取上次检查后的新视频
3. **后台异步生成**：报告生成完全异步，用户无需等待

## 六、技术细节补充

### 6.1 yupi-hot-monitor 的 Cookie 生成方式

```typescript
const buvid3 = `${crypto.randomUUID()}infoc`;
```

这个方式在 B 站是被广泛接受的，原因：
- buvid3 本质是浏览器唯一标识符
- B 站无法区分手动生成和浏览器生成的 UUID
- 多个开源项目使用此方式，未见封禁报告

### 6.2 现有代码的降级逻辑是否必要

经过分析，**3 级降级策略确实过于保守**：

| 降级层级 | 实际成功率 | 建议 |
|----------|-----------|------|
| space/acc/info | 约 85% | 保留 |
| card API | 约 10%（且返回数据不完整） | 移除 |
| 网页标题解析 | 约 3%（信息极少） | 移除 |

**建议**：保留 space API → 搜索 API 降级即可，覆盖 95%+ 场景。

**优化后**：
```python
import uuid

def _init_session() -> requests.Session:
    logger.info("初始化 B 站 Session，生成 buvid3 Cookie...")
    sess = requests.Session()
    sess.headers.update(DEFAULT_HEADERS)
    
    # 自动生成 buvid3 Cookie（参考 yupi-hot-monitor 方案）
    buvid3 = f"{uuid.uuid4()}infoc"
    sess.cookies.set("buvid3", buvid3, domain=".bilibili.com")
    logger.info("B 站 Cookie 生成成功：buvid3=%s", buvid3[:20] + "...")
    
    return sess
```

**预期收益**：每博主节省 5-15 秒

#### 步骤 3：简化 API 降级逻辑

**文件**：`backend/tracker_bilibili_adapter.py`

**修改 `get_creator_info` 函数**：
```python
def get_creator_info(url: str) -> CreatorInfo:
    """
    输入 UP 主主页 URL，返回博主信息。
    两级降级策略：
    1. space/acc/info API（需 buvid3 Cookie，无需 WBI 签名）
    2. 搜索 API（用 UP 主名搜索，按 mid 过滤）
    """
    mid = extract_mid_from_url(url)
    if not mid:
        mid = url.strip().split("/")[-1].split("?")[0]
        if not mid.isdigit():
            raise ValueError("无法从 URL 中提取 B 站 UP 主 mid，请输入正确的个人空间链接")

    session = _get_session()

    # === 方案 1: space/acc/info ===
    for attempt in range(2):  # 减少重试次数：3 次 → 2 次
        try:
            _rate_limit_wait()
            resp = session.get(
                SPACE_INFO_API,
                params={"mid": mid},
                timeout=(10, 30),
                headers=DEFAULT_HEADERS,
            )
            data = resp.json()

            if data.get("code") == 0:
                mark_request_success()
                info = data.get("data", {})
                return CreatorInfo(
                    platform="bilibili",
                    platform_id=str(mid),
                    name=info.get("name", "未知 UP 主"),
                    avatar_url=info.get("face", ""),
                    description=info.get("sign", "")[:200],
                    home_url=f"https://space.bilibili.com/{mid}",
                )

            error_code = data.get("code", -1)
            error_msg = data.get("message", "未知错误")

            if error_code == -799 or "频繁" in error_msg:
                # 限流，等待后重试
                mark_request_failure()
                wait = 5 * (attempt + 1)  # 缩短等待时间
                logger.warning("space/acc/info 限流 mid=%s, 等待%d 秒", mid, wait)
                time.sleep(wait)
                if attempt < 1:
                    _init_session()  # 刷新 Session
                    session = _get_session()
                continue

            # 其他错误，直接跳到备用方案
            logger.warning("space/acc/info 错误 code=%s msg=%s", error_code, error_msg)
            break

        except requests.RequestException as e:
            logger.warning("space/acc/info 请求异常：%s", e)
            if attempt < 1:
                time.sleep(2 * (2 ** attempt))

    # === 方案 2: 搜索 API（用 UP 主名搜索，按 mid 过滤） ===
    # 尝试从 space/acc/info 获取名字
    creator_name = ""
    try:
        _rate_limit_wait()
        resp = session.get(
            SPACE_INFO_API,
            params={"mid": mid},
            timeout=(10, 30),
            headers=DEFAULT_HEADERS,
        )
        data = resp.json()
        if data.get("code") == 0:
            creator_name = data["data"].get("name", "")
    except Exception:
        pass

    if creator_name:
        try:
            videos = _fetch_videos_via_search_api(mid, creator_name, cutoff, session)
            if videos:
                return videos
        except Exception as e:
            logger.warning("搜索 API 获取视频失败：%s", e)

    raise ValueError("获取 B 站 UP 主信息失败（2 种方案均失败），请稍后重试")
```

**移除的函数**：
- `_fetch_videos_via_space_api()` - 不再使用 space/arc/search API
- card API 降级逻辑
- 网页标题解析逻辑

**预期收益**：每博主节省 5-10 秒，代码更简洁

## 十一、总结

### 核心优化思路

1. **减少不必要的请求**：Cookie 自动生成、移除冗余降级
2. **缩短等待时间**：降低基础限流、缩短惩罚间隔
3. **简化调用链路**：保留核心降级，移除边缘降级
4. **分阶段实施**：先低风险，再中风险，最后架构优化

### 关键注意事项

1. **不要完全移除限流**：B 站对追踪场景敏感，需要保持一定克制
2. **保留降级机制**：B 站 API 不稳定，需要容错
3. **监控失败率**：优化后密切监控，如失败率显著上升需回退
4. **用户沟通**：提前告知用户优化可能带来的影响
